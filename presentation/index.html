Frontend · 1/3

¿Por qué Angular + 11ty?

Angular 21 → /menu

Panel privado: estado complejo, formularios densos, guards, signals.

Tipado fuerte y arquitectura mantenible a 3 años vista.

11ty → /

Landing, legales, blog, 404. Contenido casi 100% estático.

Una SPA aquí sería over-engineering: peor SEO y peor Lighthouse.

Cada herramienta donde mejor rinde — y se sirven juntas por el mismo Nginx.

La decisión más visible del frontend es usar dos tecnologías: Angular para lo dinámico, 11ty para lo estático. Una SPA para la landing sería matar moscas a cañonazos: SEO peor, Lighthouse peor, complejidad gratuita. Cada herramienta donde brilla.
Frontend · 2/3

SCSS compartido entre Angular y 11ty

  • Ambos compilan desde frontend/src/styles/.
  • Arquitectura ITCSS (settings → tools → generic → elements → layout → components → utilities).
  • Nomenclatura BEM — sin colisiones, sin duplicados.
  • Una sola fuente de verdad cromática para ambos frontends.
Como Angular y 11ty comparten estilos, no hay dos sistemas: hay uno solo. ITCSS marca el orden de las capas, BEM marca cómo se nombran los componentes. Mismo CSS para el panel y para la landing — coherencia gratuita.
Frontend · 3/3 · Detalle técnico

El theme switch como signal


@Injectable({ providedIn: 'root' })
export class ThemeService {
  readonly theme = signal<Theme>(this._resolveInitial());

  toggle(): void {
    this.setTheme(this.theme() === 'dark' ? 'light' : 'dark');
  }

  private _apply(theme: Theme): void {
    const root = document.documentElement;
    root.classList.toggle('dark', theme === 'dark');
    root.setAttribute('data-theme', theme);
  }
}

Una sola clase .dark en <html> reescribe toda la UI vía CSS custom properties.

El cambio de tema usa signals de Angular y toca una sola clase en el html. Como toda la paleta vive en custom properties, basta con esa clase para repintar el panel entero en una transición. Sin recargas, sin recompilar nada.
Frontend · curiosidad

Y se sincroniza entre Angular y 11ty


// El evento 'storage' SOLO se dispara en otras pestañas,
// nunca en la que hizo el setItem — exactamente lo que necesitamos
// para que la app espejo se entere del cambio sin entrar en bucle.
window.addEventListener('storage', (event: StorageEvent) => {
  if (event.key !== STORAGE_KEY) return;
  const next: Theme = event.newValue === 'dark' ? 'dark' : 'light';
  if (next !== this.theme()) {
    this.theme.set(next);
    this._apply(next);
  }
});

Si cambio el tema en el panel Angular y tengo la landing 11ty abierta en otra pestaña, también vira. Mismo origen, mismo localStorage.

Lo bonito: si cambio el tema en el panel y tengo la landing abierta en otra pestaña, la landing también cambia. El evento storage se dispara solo en las pestañas distintas a la que escribió, así que no hay bucle. Misma marca, misma percepción.

Backend · 1/3

Flask · blueprints · multi-tenant

  • Flask + Gunicorn, blueprints modulares (auth, perfil, vault, heatmap, apartamentos…).
  • ¿Por qué Flask? Microframework: control fino sobre middlewares, sin magia.
  • JWT HS256 + CSRF double-submit + rate limit + CORS por orígenes.
  • Aislamiento por empresa_id en todas las queries.
El backend es Flask. Microframework para tener control fino: yo decido qué middleware entra. Cada herramienta operativa es un blueprint independiente, y todas las consultas llevan empresa_id porque el sistema es multi-tenant de verdad.
Backend · 2/3 · "patrón DTO"

Marshmallow como frontera contractual


class ApartamentoCreateSchema(Schema):
    nombre    = fields.String(required=True, validate=validate.Length(min=1, max=200))
    direccion = fields.String(load_default=None, validate=validate.Length(max=300))
    ciudad    = fields.String(load_default=None, validate=validate.Length(max=100))

    @pre_load
    def sanitize(self, data, **kwargs):
        # Strip HTML ANTES de validar — nh3 mata cualquier tag
        for field in ("nombre", "direccion", "ciudad"):
            if field in data and isinstance(data[field], str):
                data[field] = nh3.clean(data[field], tags=set()).strip()
        return data

Sanitiza primero, valida después. El blueprint nunca toca un dict crudo del cliente.

No se llama DTO en el código, pero cumple ese rol: cada endpoint tiene su schema de Marshmallow que sanitiza primero con nh3 y valida después. El blueprint nunca recibe un dict crudo de Flask, recibe un objeto ya limpio. Frontera contractual entre HTTP y negocio.
Backend · 3/3 · Curiosidad

Datos de huéspedes: jamás en BD

Cifrado simétrico Fernet

Las API keys del PMS y del proveedor de IA viven en BD cifradas con clave maestra.

FERNET_KEY fuera del repo, en el entorno.

Huéspedes en memoria

El mapa de calor y la sincronización Google trabajan con datos en RAM.

Cuando termina la petición, se evaporan. Cero PII persistida.

RGPD no como póster en la pared — como decisión arquitectónica.

Y la decisión más curiosa del backend: los datos de huéspedes nunca tocan la base de datos. Se procesan en memoria, se sirve la respuesta, se evaporan. RGPD por diseño, no por documento. Y lo poco sensible que persiste, como las API keys, va cifrado con Fernet.
DevOps · 1/3 · El detalle que no se ve

En realidad hay dos Nginx

Nginx externo · ./nginx

Reverse proxy. Decide qué prefijo va a qué contenedor.

Aplica cabeceras de seguridad globales (CSP, HSTS, X-Frame-Options).

Nginx interno · en cada contenedor estático

Sirve el dist/ de Angular y el _site/ de 11ty.

Resuelve el SPA-fallback de Angular y los trailing slashes de 11ty.

Externo reparte tráfico. Internos sirven archivos. Roles diferenciados, mismo binario.

Algo que la gente suele pasar por alto: hay dos Nginx. Uno externo que es el reverse proxy y aplica las cabeceras de seguridad. Y otro interno dentro de los contenedores estáticos, sirviendo el dist y resolviendo el SPA fallback. Mismo binario, dos responsabilidades.
DevOps · 2/3 · Caso real de debugging

CSP estricta vs. scripts del 11ty


add_header Content-Security-Policy "
  default-src 'self';
  script-src 'self' https://challenges.cloudflare.com
             'sha256-Ig5A+5cewHBFUnM/P9kWkADehgoLk1wG8ReLwvNfLGQ='
             'sha256-eJGI0Ik4oYe/PKLDOt4wcN76wYs8h+Ew05pMzdY6xG8=';
  style-src 'self' https://fonts.googleapis.com 'unsafe-inline';
  ...
  object-src 'none'; base-uri 'self'; form-action 'self';
" always;
  • El 11ty inyecta scripts inline mínimos (toggle de tema, anti-FOUC).
  • Opción A — 'unsafe-inline'. Rechazada: rompe el modelo de amenaza.
  • Opción B — hashear cada script y declararlo en la CSP. Hecho.
La CSP estricta y el 11ty chocaron de frente. El 11ty inyecta dos scripts inline para el tema y para evitar el flash de contenido. Podía abrir la puerta con unsafe-inline, pero eso rompe la CSP. La solución: hashear los scripts en sha256 y meterlos en la policy.
DevOps · 3/3

Railway · DNS interno que cambia cada deploy


# DNS resolver interno de Railway — TTL 1s
# evita IPs caducadas tras redeploys
resolver [fd12::10] ipv6=on valid=1s;

set $backend_url http://backend-sidekick.railway.internal:${BACKEND_PORT};

location /api/ {
    proxy_pass $backend_url;   # variable -> re-resolución por petición
    ...
}

TLS lo termina Railway en el edge — Nginx solo escucha en :80 dentro de la red privada IPv6.

En Railway hay otro reto: cada redeploy cambia las IPs internas. Si Nginx las cachea, todo cae. La solución es el resolver IPv6 con TTL de un segundo y usar variables en proxy_pass para forzar la re-resolución por petición. TLS lo termina Railway, nosotros vamos en HTTP.
